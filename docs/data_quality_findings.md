# Аудит качества данных M.Video — следы человеческой сборки

Ручная проверка трёх файлов на неточности, которые появляются, когда датасет
собирает человек: перепутанные колонки, транслит, дубли, обрезанные строки.

Проверено на семпле **600 000 строк** `query_clicks` (seed=7), полном
`sku_desc` (row group, ~123k) и всём каталоге `skus.pkl` (546k офферов).

---

## 1. query_clicks.parquet

Схема: `sku_id, sku_name, sku_brand_name, sku_price, sku_subject_id, sku_seo_id, query_text, sku_position`.
Формальных `NaN` нет, но много **пустых строк `""`** — это «дыры», а не честные значения.

### 1.1 Потерянные карточки (главная проблема)
- **28.3%** строк — пустой `sku_brand_name`.
- **21.7%** строк — пустой `sku_name`.
- **2176** строк с `sku_price <= 0`, и в **~96%** из них одновременно пусты и название, и бренд.

Вывод: это клики, у которых атрибуты SKU не подтянулись при сборке (join не
срослось). Клик зафиксирован, а карточка — нет. Для обучения такие строки —
шум: бренд/категорию из них не достать.

### 1.2 Перепутанные колонки: артикул уехал в поле бренда
- **66** строк, где `sku_brand_name` состоит **только из цифр** (`27942`, `27919`, `360`, `0`, `211`, `4088`…).
- В **98.5%** этих строк то же число присутствует внутри `sku_name`.

Пример:
```
sku_name       = "Вентилятор напольный 27942 GL 8112"
sku_brand_name = "27942"     <- это артикул из названия, а не бренд
```
Классический сдвиг колонок / неверный парсинг: там, где бренда не было, в поле
затянулся числовой код из названия.

### 1.3 Аномалии позиции клика
- `sku_position`: min 0, max **4080**, медиана 4.
- **23.8%** строк имеют `position == 0` — скорее «нет данных о позиции», а не
  «нулевая позиция». Это надо учитывать, если использовать позицию как сигнал
  качества пары query↔SKU.

### 1.4 Транслит и смесь языков
- **47 583** уникальных запроса (из 135 940) содержат **и кириллицу, и латиницу**
  одновременно: `iPhone 17 про Макс 512`, `bosh лодильник двухдверны`,
  `стиральная машинка hisense`, `HONOR часы`.
- Явный транслит русских слов латиницей: `Stiralnay mashina`, `noutbuk`,
  `televizor`, `kholodilnik`. Модель обязана быть устойчивой к смешанной
  раскладке.
- **Забытая раскладка (EN вместо RU): 80+ запросов.** Человек печатал по-русски,
  не переключив раскладку, — получилась латинская «кракозябра», которая при
  обратном отображении QWERTY→ЙЦУКЕН читается по-русски:
  - `dfhjxyfz gfytkm ,tkjt cntrkh` → «варочная панель белое стекло»
  - `dcnhfbdftvfz cnbhfkmyfz vfibyf` → «встраиваемая стиральная машина»
  - `gsktcjc dthnbrfkmysq ghjdjlyjq` → «пылесос вертикальный проводной»
  - `pfhzlyjt ecnhjqcndj czjvb` → «зарядное устройство сяоми»

  Такой запрос без авто-исправления не находит вообще ничего → 100% потеря клика.
  Детект дешёвый (одна таблица маппинга), место — «голова» каскада рядом с regex.
  Полный список: `artifacts/layout_translit_final.json`.
- В `sku_brand_name` — **28** брендов со смесью алфавитов:
  `Dyson Фен`, `Фаzа`, `Dомашний`, `ULTRAМARIN`, `ГрунтТeх`, `мakPro`,
  `Ресантa` (последняя `а` — кириллическая → омоглиф!).

### 1.5 Дубли брендов из-за регистра/раскладки
- **205** групп брендов, отличающихся только регистром/пробелами:
  `APPLE`/`Apple`, `SAMSUNG`/`Samsung`, `HONOR`/`Honor`/`honor`,
  `Lenovo`/`lenovo`, `INDESIT`/`Indesit`…
- **~209** групп-кандидатов на **омоглифы** (кириллические буквы, визуально
  идентичные латинским: `а е о р с х у…`). Без нормализации это раздувает
  словарь брендов и путает матчинг.

### 1.6 Прочее
- **41 715** полных дублей строк в семпле (одинаковый клик записан несколько раз).
- **8 900** строк, где `query_text` дословно равен бренду (не ошибка, но
  полезно для brand-классификатора).
- **474** уникальные пары (0.6%), где `sku_brand_name` вообще не встречается в
  тексте `sku_name`: `Холодильник Stinol …` при бренде `Стинол` (перевод/транслит),
  `Онлайн-кинотеатр KION …` при бренде `КИОН`, и т.п.

## 2. sku_desc.parquet

Схема: `sku_id, title, description` (1 177 200 строк).

- **35 810** `title` заканчиваются на висячий дефис `-` — обрезанное название,
  где потерялась хвостовая часть (обычно бренд/модель):
  `Набор отверток GSS -`, `Робот GSS -`, `Ключница GSS -`.
- **44 282** дубля `title`.
- **19** строк, где `title == description` (описание не заполнено, скопирован title).
- **10** строк с `description` короче 10 символов.

## 3. skus.pkl (YML-каталог)

Структура оффера: `@id, @available, vendor, model, name, categories, param`
(6 963 категории, 546 069 офферов). Дублей id нет.

Важные структурные факты (отличаются от исходного предположения команды):
- Категория оффера лежит в поле **`categories`** (список), а **не** `categoryId`.
  Поэтому «нет categoryId» у всех офферов — это не порча данных, а иная схема.
- В оффере **нет поля `price`** — цена есть только в `query_clicks`.
- **116** офферов с числовым `vendor`, **11** — с пустым, **229** `name`
  заканчиваются на висячий `-` (та же обрезка, что и в sku_desc).

## Что из этого следует для решения

1. **Нормализация — обязательный слой.** Регистр, транслит, омоглифы,
   схлопывание дублей брендов (`APPLE`=`Apple`, `Стинол`=`Stinol`) до
   каноничной формы. Без этого словарь брендов раздут и «дырявый».
2. **Фильтрация мусора перед обучением.** Строки с пустыми name/brand и
   `price<=0` (потерянные карточки) — выкидывать или помечать, чтобы не учить
   модель на дырах.
3. **Артикул ≠ бренд.** Правило: числовой `sku_brand_name` → не бренд, а шум;
   искать реальный бренд в названии.
4. **Позиция 0 = пропуск.** Трактовать `position==0` как отсутствие сигнала.
5. **Устойчивость к смешанной раскладке** — прямое требование к NER: 35% с
   лишним запросов смешанные (лат+кир).

Скрипт аудита: `scripts/audit_data_quality.py`.
